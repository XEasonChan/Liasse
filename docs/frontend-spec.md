# 前端需求文档（v0，待 review）

基于 2026-05-18 提供的 UI mockup（Liasse v1.0.0）整理。原始 mockup 包含一些不必要的侧栏项，本文档已删减。文末「决策点」一节列出仍需用户确认的取舍。

## 1. 产品定位

非开发者的本地访谈转录与分析工具。三步流程：

1. 拖拽音频文件（或文件夹）进来
2. 后台按队列跑 Qwen3-ASR + 可选发言人识别 + 可选总结
3. 点开任意一条已完成任务，看逐字稿 / 总结 / 用 LLM 问答

所有计算与模型全部本地（M1/M2/M3/M4 + 已下载模型），不联网。

## 2. 信息架构（侧栏精简后）

```
Liasse v1.0.0
├─ 转录            ← 主工作台（上传 + 任务列表）
├─ 设置
└─ [模型状态面板]   ← 侧栏底部固定，显示已加载模型
```

**删除的原侧栏项及理由：**

| 原项 | 处理 | 理由 |
|---|---|---|
| 发言人识别 | 删除 → 改为上传时的 toggle | 不是一个独立"页面"，是每个任务的可选参数 |
| 自动分段 | 删除 → 自动行为，不暴露开关 | ASR/diarization 本身就分段，没必要单独成项 |
| 总结 (Qwen 8B) | **待决策**（见决策点 D1） | 是否需要独立"总结库"视图 |
| AI Chat (Qwen 8B) | **待决策**（见决策点 D1） | 是否需要脱离任务上下文的纯聊天 |

## 3. 页面 A：转录主页

### 3.1 顶部区域

- 标题：「转录」
- 副标题：「上传音频文件或选择包含音频文件的文件夹，自动转录并识别发言人，生成分段结果。」

### 3.2 上传区

中间一块拖拽热区，下方两个按钮：

- `选择文件`（multi-select）
- `选择文件夹`（自动递归扫描，但只扫一层子目录，避免误吸）

支持格式：`mp3 / wav / m4a / flac / aac / ogg / wma / mp4`（音视频均接受，视频走 ffmpeg 拆音轨）。

### 3.3 上传区下方 toggle 行（取代被删的侧栏项）

新增。文件加入队列**前**让用户配置本次（或本批）任务参数：

```
┌──────────────────────────────────────────────────────────┐
│ □ 发言人识别  人数: [自动 ▾]  □ 生成总结                 │
│ ASR 模型: ( ●0.6B 快速  ○1.7B 高质量 )  语言: [中文 ▾]   │
└──────────────────────────────────────────────────────────┘
```

字段说明：

| 字段 | 默认 | 说明 |
|---|---|---|
| 发言人识别 | ON | 触发 pyannote 流程，~30% 额外耗时 |
| 人数 | 自动 | 选项：自动 / 2 / 3 / 4 / 5+。访谈研究通常 2 |
| 生成总结 | ON | 触发 Qwen3:8b 分块总结，需要 ollama 在跑 |
| ASR 模型 | 0.6B | 0.6B 大约实时 0.67x；1.7B 慢 ~50%，质量更高 |
| 语言 | 中文 | 中文 / 英文 / 粤语 / 自动 |

**重要**：这些 toggle 的值在"添加到队列"那一刻被锁定到任务对象上。后续任务列表里每一行展示自己的参数标签。

### 3.4 任务列表

每一行：

```
┌─文件图标─┬────────────────────────┬──进度─┬─状态─┬──时间──┬──操作──┐
│   📄    │ 会议记录_2024-05-20.mp3 │ 65%   │ 转录中 │ 14:30  │ ■  📁  │
│         │ 25.4 MB · 00:45:12      │       │       │        │        │
└─────────┴────────────────────────┴───────┴───────┴────────┴────────┘
```

- 整行可点击 → 打开「转录详情页」（已完成任务才可点；进行中只能查看实时进度）
- 操作列：
  - 进行中：停止 ■ / 查看日志 📋
  - 已完成：打开输出目录 📁 / 删除 🗑
  - 失败：重试 ↻ / 查看错误 ⚠ / 删除 🗑
- 右上「清空已完成」按钮（注：原 mockup 写的是「清空 completed」，中英混排，建议改）

### 3.5 底部状态栏

- 左侧：「后台转录进程」+ 当前正在处理的文件 + 模型名
- 右侧：CPU / RAM / GPU 占用 mini-sparkline + 「打开日志」按钮

**待 review**：底部状态栏在 MVP 里是否必要？详见决策点 D2。

## 4. 页面 B：转录详情页（点击任务后）

整个页面分三栏：左侧返回栏（窄）+ 中央内容（宽）+ 右侧 Q&A 侧栏（中）。也可以是「左侧 = 全局导航不变」+「中央 + 右侧」。

### 4.1 顶部信息条

```
← 返回   会议记录_2024-05-20.mp3              [导出 ▾] [删除]
         25.4 MB · 00:45:12 · 2 个发言人 · 0.6B · 中文
         转录于 2024-05-20 15:23  耗时 33:42
```

导出菜单：Markdown / JSON / SRT / 全部打包 zip。

### 4.2 中央 Tab 视图

```
[ 逐字稿 ] [ 总结 ] [ 时间线 ]   ← 3 个 tab
```

#### Tab 1: 逐字稿

按发言人分块（已经是 `transcript.md` 的格式）：

```
SPEAKER_00   00:00:05 - 00:00:07
hello 大家好我是小俊

SPEAKER_01   00:00:30 - 00:00:52
二零二四年加入 Anthropic...
```

每一段：

- 鼠标悬停高亮，点击 → 跳到音频对应位置（**待 review**：是否带音频播放器，见 D3）
- 段落内文字**可编辑**（用于修正 ASR 错误）
- 发言人标签**可点击改名**（SPEAKER_00 → "主持人"），改了后同一发言人全局生效
- 编辑过的段落带 ✎ 标记，导出时按用户修改后的版本走

#### Tab 2: 总结

如果该任务勾了"生成总结"：

- 总结正文（Markdown 渲染）
- 顶部一个「重新生成总结」按钮（带 prompt 编辑入口，方便用户调整角度）
- 显示用的是哪个总结模型（Qwen3:8b）+ 用了多少 token

如果没勾：显示空状态 + 按钮「现在生成总结」。

#### Tab 3: 时间线（可选）

可视化展示发言人切换 + 关键词热点。**待 review**：MVP 是否需要，见 D4。

### 4.3 右侧 Q&A 侧栏（Qwen3:8b）

针对当前这段访谈的 chat：

```
┌──────────────────┐
│ 询问这段访谈      │
│ ─────────────── │
│                  │
│ 对话历史...      │
│                  │
│ ─────────────── │
│ [输入框    发送] │
│ 用 Qwen3:8b 本地 │
└──────────────────┘
```

关键设计：

- LLM 调用时把**当前任务的完整逐字稿**作为 system context（或 RAG，看长度，3-4 小时录音的转录可能超过 8B 模型上下文，需要分块召回——**待 review**，见 D5）
- 对话**只跟当前任务绑定**，切到另一个任务就是新的 chat
- 历史对话保存在任务对象里，重新打开该任务能看到之前问过什么

## 5. 页面 C：设置

最小集：

- **模型管理**
  - ASR 模型：当前已下载列表（0.6B ✓ / 1.7B 未下载 [下载]）+ 缓存位置 + 缓存大小
  - 总结/Chat 模型：Ollama 中已有的模型 + 切换
  - pyannote：状态 + 缓存大小
- **输出目录**：默认 `~/Documents/Liasse/`，可改
- **性能**：CPU 核数限制 / GPU 用与不用
- **隐私 / 数据**
  - 完全离线开关（强制 `HF_HUB_OFFLINE=1`）
  - 一键清空所有任务历史
- **关于**：版本 / 许可证 / 模型来源

## 6. 数据模型（前端契约）

每个任务对象大致：

```ts
type Task = {
  id: string;
  audioPath: string;        // 用户原文件，App 不复制
  fileName: string;
  fileSizeBytes: number;
  durationSec: number;

  status: "queued" | "running" | "done" | "failed" | "stopped";
  progress: number;         // 0-1
  progressStage?: string;   // "正在转录" / "正在识别发言人" / "正在生成总结"
  errorMessage?: string;

  // 任务创建时锁定的参数
  config: {
    asrModel: "Qwen/Qwen3-ASR-0.6B" | "Qwen/Qwen3-ASR-1.7B";
    language: "Chinese" | "English" | "Cantonese" | "auto";
    diarize: boolean;
    numSpeakers: number | null;   // null = auto
    summarize: boolean;
    summaryModel: "qwen3:8b";
  };

  createdAt: string;        // ISO
  completedAt?: string;
  elapsedSec?: number;

  outputs?: {
    markdownPath: string;
    jsonPath: string;
    srtPath: string | null;
  };

  // 用户编辑的覆盖层（不动原始 ASR 输出，前端 merge）
  edits?: {
    speakerLabels: Record<string, string>;  // "SPEAKER_00" -> "主持人"
    segmentOverrides: Record<string, string>; // segmentId -> 修正后文字
  };

  // 该任务关联的 chat
  chatMessages?: Array<{ role: "user" | "assistant"; content: string; ts: string }>;
};
```

任务列表 = 本地 SQLite 持久化（或 JSON 文件），App 重启后状态保留。

## 7. 任务状态机

```
queued → running ──┬──→ done
                   ├──→ failed (有错误信息)
                   └──→ stopped (用户主动停)
```

一次只跑一个 `running`（避免 8GB/16GB 内存争抢）。其他停在 `queued`。

## 8. 决策点 — 待 review

请你过一遍这几个，标 "保留 / 删除 / 改为 X"：

### D1. 侧栏到底留哪几项？

当前候选三种方案：

| 方案 | 侧栏内容 | 优点 | 缺点 |
|---|---|---|---|
| A. 极简 | 转录、设置 | 干净，每个任务自带总结+chat | 没法跨任务做检索 / 全局聊天 |
| B. 折中 | 转录、设置、AI Chat | 保留一个全局 chat 入口（可以选多个任务做参考） | 概念重叠：详情页里也有 chat |
| C. 完整 | 转录、总结、AI Chat、设置 | 跟原 mockup 接近 | "总结"页含义不清，容易跟详情页 tab 撞 |

**我倾向 A**：研究场景用户进来基本都是为了"看某一段访谈"，没必要在侧栏暴露"全局总结/全局聊天"。如果之后需要再加。

### D2. 底部状态栏（CPU/RAM/GPU + 日志按钮）

MVP 是否要？我倾向**简化**到一行小字「正在处理 xxx.mp3（45/60 分钟）」+「打开日志」即可，CPU/GPU 监控对用户决策意义不大，反而占视觉空间。

### D3. 详情页要不要带音频播放器？

如果要：点逐字稿任意一段 → 跳转音频到对应时间点 → 边听边校对。这是访谈研究里**很有用**的功能，但实现复杂度不小（要 waveform、要时间戳精确同步）。

倾向：**MVP 第一版不做**，只做编辑文字。第二版加。

### D4. 时间线 tab

发言人切换可视化（甘特图风格）+ 关键词热点。**MVP 不做**，先把逐字稿和总结打磨好。

### D5. Q&A 怎么塞 3-4 小时的上下文

Qwen3:8b 的 context window 一般 32K-128K tokens。3-4 小时转录中文 ≈ 30K-60K 字 ≈ 60K-120K tokens，可能塞不下完整 prompt。

可选方案（按推荐顺序）：

1. **简单分块召回**：把转录切 1000 字/块向量化（小 embedding 模型），用户问问题时召回 top-5 块拼到 prompt 里。需要本地 embedding 模型，~100MB。
2. **暴力裁剪**：直接把转录截到上下文窗口剩余位置塞进去。中文访谈可能直接被截掉一半。
3. **第一次问的时候，先用 Qwen3:8b 把转录预压缩成结构化要点**（人物/事件/观点），后续 Q&A 都跟这个压缩版聊。压缩一次缓存住。

倾向 **3**，因为简单、不引入新依赖、对学术研究场景"快速问几个问题"够用。需要更深度时再上 1。

### D6. 关于"清空 completed"按钮

原 mockup 是中英混排。统一改成 **「清空已完成」**。

### D7. 设置里要不要加"完全离线"开关

我倾向加：勾上后强制 `HF_HUB_OFFLINE=1`，App 不再尝试任何外网请求（包括检查更新）。这对 IRB 合规场景很重要——心理上的"我可证明不可能上云"。

### D8. 删除任务时，是否同时删除磁盘上的输出文件？

二选一弹窗：「仅从列表移除」/「同时删除转录输出」。原始音频不动。

## 9. 不在 MVP 范围内（明确不做）

- 多语言 UI（先只做中文界面）
- 用户账号、协作分享
- 任务批量导出为单一报告
- 与 NVivo / Atlas.ti 等定性研究软件的直接集成（先靠 JSON 导出）
- 实时录音转录（先做文件转录）
- 音频前处理（降噪 / 增益）
